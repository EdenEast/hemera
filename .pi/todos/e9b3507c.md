## Problem Statement

The user wants Hemera to become a working homelab platform that can host internal self-hosted services and demonstrate practical DevOps experience. The current repository has goals, glossary language, an Initial Setup design document, ADRs for NixOS-based Node Configuration and k3s, plus a placeholder repo skeleton, but the Terraform, NixOS, secret-management, template, deployment, and validation workflows are not fully implemented yet.

## Solution

Implement the Hemera Initial Setup so that Thor runs Proxmox, Terraform provisions three NixOS Cluster Nodes from a reusable Proxmox VM template, NixOS Node Configuration is applied remotely with `nixos-rebuild --target-host`, and k3s forms a usable one-control-plane/two-worker Kubernetes cluster. The Initial Setup is complete when the cluster is Ready, local-path storage can provision a test volume, and the k3s default Traefik/ServiceLB path can expose a sample app to the LAN.

## Implementation Decisions

- Build or complete a Terraform Proxmox infrastructure module that declares the Cluster Node VM topology and uses the `bpg/proxmox` provider.
- Keep Terraform responsible only for VM lifecycle: cloning the template, VM identity in Proxmox, CPU, memory, disks, and network attachment.
- Keep Terraform out of k3s installation, in-cluster Kubernetes resource management, and OS-level Node Configuration.
- Use local Terraform state for Initial Setup, keep state out of git, and document manual backup to TrueNAS.
- Build or complete a NixOS host configuration layer with a shared common module and role-specific k3s server/agent modules.
- Treat the NixOS host configuration layer as a deep module: its interface is the host role, hostname, static networking values, and secret references; it encapsulates SSH, QEMU guest agent, Nix settings, k3s service configuration, and sops-nix wiring.
- Build or complete a secrets module around sops-nix for the k3s cluster token.
- Treat secret management as a deep module: its interface is named secrets exposed to NixOS services; it encapsulates recipient configuration, host SSH-key-derived identities, and runtime secret file paths.
- Generate a strong k3s cluster token once and store it only as encrypted secret material.
- Use one admin age key and one host age identity per NixOS host.
- Derive host age identities from unique SSH host keys.
- Ensure the NixOS Proxmox template does not include cloned SSH host keys or cloned machine identity.
- Prefer generating the NixOS Proxmox template with Nix tooling; use an official NixOS cloud image as the fallback if direct generation is not feasible.
- Use one control-plane Cluster Node and two worker Cluster Nodes for Initial Setup.
- Use the conservative VM resource profile: 2 vCPU per node, 3GB RAM per node, 40GB disk for the control plane, and 60GB disk per worker.
- Use static IP addresses configured by NixOS; initial placeholder values are in the `192.168.1.0/24` network and must be confirmed during implementation.
- Use k3s defaults for Initial Setup service exposure and storage: local-path provisioner, Traefik, and ServiceLB.
- Build or complete operational scripts for single-node deployment, full-cluster deployment, and kubeconfig retrieval.
- Treat the deployment script layer as a deep module: its interface is a small set of stable commands; it encapsulates host target resolution, rebuild command construction, and kubeconfig rewriting.
- Keep generated outputs, Terraform variable files, Terraform state, decrypted secrets, age private keys, and local kubeconfigs out of git.
- Keep the single Initial Setup document updated as both design and runbook.
- Respect ADR 0001: NixOS owns Node Configuration and `nixos-rebuild --target-host` is the Initial Setup deployment mechanism.
- Respect ADR 0002: k3s is the Initial Setup Kubernetes distribution.

## Testing Decisions

- Good tests should validate external behavior and operational outcomes, not internal implementation details. For this repository, that means validating generated plans, NixOS evaluations, script behavior, and cluster readiness rather than testing individual lines of configuration.
- Test the Terraform module by running formatting, validation, and a plan against example or non-secret inputs where possible.
- Test the NixOS configuration layer by checking that each host configuration evaluates successfully.
- Test the secrets module by checking that encrypted secret metadata is present, plaintext secrets are absent from committed files, and NixOS can resolve the expected secret names during evaluation.
- Test deployment scripts with shell linting or dry-run-friendly checks where possible, especially argument validation and known-node handling.
- Test the k3s cluster externally after deployment by verifying all Cluster Nodes are Ready through Kubernetes API output.
- Test local-path storage externally by creating a PVC and confirming it binds.
- Test service exposure externally by deploying a sample app and confirming it is reachable from the LAN through k3s default ingress/load-balancing.
- Test template identity manually or with a validation script by confirming cloned Cluster Nodes do not share SSH host keys or machine-id.
- Current repository prior art is limited; there are no existing automated tests. Initial testing should therefore prioritize tool-native checks and smoke-test runbooks.

## Out of Scope

- Highly available Kubernetes control plane.
- Multiple physical Kubernetes hosts.
- TrueNAS-backed Kubernetes dynamic storage.
- Monitoring, dashboards, and alerting.
- Helmfile-managed add-ons beyond preserving room for later use.
- GitOps with Argo CD or Flux.
- Colmena-based NixOS deployment.
- Cilium or another advanced Kubernetes networking stack.
- Production-grade secret rotation workflows beyond initial sops-nix encrypted token management.
- Cloud-hosted infrastructure, Terraform Cloud, or cloud object storage.
- Full automation of Proxmox installation on Thor.
