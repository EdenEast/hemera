Added template registration automation:

- New script: `scripts/register-proxmox-template`
- Usage:

```sh
scripts/register-proxmox-template 9000 nixos-template-v1
scripts/register-proxmox-template 9001 nixos-template-v2
```

What it does:

1. Builds `.#proxmox-template`.
2. Finds `result/vzdump-qemu-*.vma.zst`.
3. Creates `/var/lib/vz/dump` on Proxmox if needed.
4. Uploads the archive to Proxmox with `scp`.
5. Runs `qmrestore` to the requested VM/template ID.
6. Sets name, QEMU guest agent, and `vmbr0` NIC.
7. Converts the restored VM to a Proxmox template.

Defaults:

```text
HEMERA_PROXMOX_HOST=root@192.168.2.80
HEMERA_TEMPLATE_BUILD_ATTR=.#proxmox-template
HEMERA_TEMPLATE_DUMP_DIR=/var/lib/vz/dump
HEMERA_TEMPLATE_STORAGE=local-lvm
HEMERA_TEMPLATE_BRIDGE=vmbr0
```

Safety:

- If the VM/template ID already exists, it fails by default.
- To intentionally replace an existing ID, set `HEMERA_TEMPLATE_REPLACE=1`.
- Recommended workflow remains versioned template IDs, e.g. `9000`, `9001`, `9002`.

Docs updated in `docs/nixos-proxmox-template.md` to prefer the script.

Network issue after boot:

User reported cloned VMs now boot to NixOS login, but `ssh-keyscan` returns no keys and VM console shows no network information.

Cause: The upstream Proxmox image module enables cloud-init and sets NixOS networking to wait for cloud-init network data. Our Terraform resource was previously ignoring/providing no `initialization` block, so the VM booted but had no configured network.

Fix added:

- `terraform/proxmox/main.tf`
  - added `started = true`
  - added `initialization` block with:
    - `datastore_id = var.proxmox_datastore_id`
    - `interface = "ide2"`
    - DNS server `192.168.2.1`
    - per-node static IPv4 `${each.value.ip}/24`
    - gateway `192.168.2.1`

This uses Proxmox cloud-init to seed the same static IPs that NixOS later declares permanently. It slightly relaxes the original “Terraform does not own OS network” boundary for bootstrap only, but is practical with the upstream Proxmox image module.
